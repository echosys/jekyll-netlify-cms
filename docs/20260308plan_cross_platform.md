## Cross-Platform Strategy

### Why NOT a Single React App Wrapped in a Desktop Shell?

Using Electron or a Python webview to wrap the React web app on the desktop seems
attractive at first, but it breaks down for this project because:

- **Local storage won't work the same way.** A Python-hosted webview (e.g. PyWebView
  or Qt WebEngine) serves the frontend from a local file path or localhost. The React
  app's `localStorage`, `IndexedDB`, and `File` APIs behave differently inside a
  sandboxed webview vs. a real browser — especially around file system access and
  saving files to custom directory paths.
- **File I/O is fundamentally different.** The desktop app needs to read/write zip
  files, rename image files, and manage a folder hierarchy next to the executable.
  This is easy in Python/PyQt but requires complex bridge code (IPC) when wrapping a
  browser-based app.
- **No meaningful code reuse saves.** The UI framework differences between React and
  PyQt mean you'd maintain two full UI codebases anyway. The bridge overhead is not
  worth it.

**Decision: Native per platform, shared JSON schema as the contract.**

| Platform        | Stack                       | Rationale                                             |
|-----------------|-----------------------------|-------------------------------------------------------|
| macOS / Windows | Python + PyQt6              | Full file system access, native feel, pip-installable |
| Web             | React + ReactFlow + Node.js | Best graph UI library, real browser APIs              |
| Android / iOS   | Flutter (Dart)              | Single codebase for both mobile platforms             |

The **shared JSON schema + zip format** is the cross-platform contract. Any platform
can export a `.zip` that any other platform can import.

## Local Folder Structure

Saved next to the application — **not** in system folders (not `~/Library`, not `AppData`).

```
FamilyTrees/                        ← parent folder alongside app
  smith_family/
    tree.json
    resources/
      1950-01-01-john_smith.jpg
      1975-06-12-jane_doe.jpg
      1980-03-01-unknown-a3f9b.jpg  ← untagged placeholder
  jones_family/
    tree.json
    resources/
```

**Image filename convention:**
```
{date}-{firstname}_{lastname}.jpg           → tagged, unique
{date}-{firstname}_{lastname}-{5id}.jpg     → tagged, duplicate name
{date}-untagged-{5id}.jpg                   → uploaded but not yet tagged
```
Rename is triggered **after** tagging is saved, not on upload.

## PostgreSQL Export/Import

### Why no `CREATE EXTENSION pgcrypto`?

`pgcrypto` was originally listed to enable `gen_random_uuid()` in SQL. However,
installing extensions requires superuser / `CREATE` privilege, which a typical
application-level Postgres user will not have.

**Solution:** UUID generation happens entirely in the **application layer**
(Python `uuid.uuid4()`, JavaScript `crypto.randomUUID()`, Dart `Uuid().v4()`).
The database stores the UUID string the app provides — no Postgres extensions required.

### DDL — No Extensions Needed

```sql
-- Family Tree PostgreSQL Schema v1.0
-- No extensions required. UUIDs generated by the application.
-- Apply with any Postgres user that has CREATE TABLE privilege on the target database.

CREATE TABLE IF NOT EXISTS family_trees (
    id              VARCHAR(36)  NOT NULL DEFAULT '',
    tree_name       VARCHAR(255) NOT NULL,
    record_type     VARCHAR(50)  NOT NULL,
    -- 'tree_meta' | 'node' | 'edge' | 'resource'
    record_id       VARCHAR(255) NOT NULL,
    tree_version    VARCHAR(20)  DEFAULT '1.0',
    payload         JSONB        NOT NULL,
    image_data      TEXT,
    -- base64 JPEG <= 2 MB, NULL for non-resource rows
    created_at      TIMESTAMPTZ  DEFAULT NOW(),
    updated_at      TIMESTAMPTZ  DEFAULT NOW(),

    CONSTRAINT pk_family_trees PRIMARY KEY (tree_name, record_type, record_id),
    CONSTRAINT uq_id UNIQUE (id)
);

CREATE INDEX IF NOT EXISTS idx_ft_tree_name   ON family_trees (tree_name);
CREATE INDEX IF NOT EXISTS idx_ft_record_type ON family_trees (tree_name, record_type);
CREATE INDEX IF NOT EXISTS idx_ft_payload_gin ON family_trees USING GIN (payload);

CREATE OR REPLACE FUNCTION update_updated_at_column()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = NOW();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

DROP TRIGGER IF EXISTS trg_family_trees_updated_at ON family_trees;
CREATE TRIGGER trg_family_trees_updated_at
    BEFORE UPDATE ON family_trees
    FOR EACH ROW
    EXECUTE FUNCTION update_updated_at_column();
```

### Export (Upsert)
- All UUIDs generated locally by the app before writing to DB.
- Each row: `INSERT ... ON CONFLICT (tree_name, record_type, record_id) DO UPDATE SET ...`
- Images compressed to ≤2 MB JPEG, then base64-encoded into `image_data`.
- Default save is **local folder**; Postgres is an optional export target.

### Import (Read)
1. Query distinct `tree_name` values — show list in UI.
2. User selects a tree (optionally deletes with type-to-confirm dialog).
3. Fetch all rows for selected `tree_name`, reconstruct `tree.json` + resource files.

## Shared Collaboration (Web Only)

```
Client A edits node
  → emits  'node:update' { tree_id, node }
  → Server broadcasts patch to all clients in room
  → Client B applies patch to local state
```

- Share URL: `https://yourapp.com/tree/{tree_id}?token=xyz`
- Server holds working copy in memory + periodic postgres/disk flush.
- Conflict resolution: last-write-wins per node/edge.
